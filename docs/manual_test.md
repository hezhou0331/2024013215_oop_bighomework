# 人工交互测试记录（Manual Test Procedures）

本文档描述通过 ConsoleUI 对项目管理功能的人工验证流程。这些测试补充自动化 PPM 格式校验测试，覆盖人机交互层的功能正确性。

## 测试环境

- 编译：`mingw32-make`
- 运行：`project_scheduler.exe`
- 平台：Windows 11 Pro（可选：Linux/macOS 通过 WSL 验证）

---

## 测试 1：新建项目与添加任务

**步骤：**

1. 启动程序，选择菜单 `1` (Create new project)
2. 输入项目名称（例如 `MyProject`）
3. 选择菜单 `4` (Add task)
4. 输入任务名称（例如 `Analysis`）、工期（例如 `3` 天）
5. 再次选择菜单 `4`，添加第二个任务 `Design`，工期 `5` 天
6. 选择菜单 `18` (Show statistics) 验证任务数量

**预期结果：**

- 项目名称正确保存
- 两个任务显示在统计中
- 任务名称与工期准确无误

---

## 测试 2：里程碑任务与工期 0 转换

**步骤：**

1. 在测试 1 的基础上继续
2. 选择菜单 `6` (Update task) 
3. 选择任务 0（`Analysis`），修改名称为 `AnalysisPhase`、工期改为 `0`
4. 选择菜单 `8` (List task relations) 选择任务 0
5. 确认任务已转为里程碑（工期显示为 0）

**预期结果：**

- 工期修改为 0 后，任务类型自动转为里程碑
- 任务下标、依赖关系保持不变
- 里程碑不能分配资源（见测试 5）

---

## 测试 3：添加依赖关系与环路检测

**步骤：**

1. 在测试 2 的基础上，假设已有两个任务
2. 选择菜单 `9` (Add dependency)
3. 前置任务：0（AnalysisPhase），后置任务：1（Design），类型 `FS`，Lag `0`
4. 依赖添加成功后，尝试添加相反方向的依赖：前置 1，后置 0
5. 系统应拒绝，显示环路检测错误

**预期结果：**

- 正向依赖正常添加
- 逆向依赖被拒绝，错误信息："Cycle detected" 或 "Dependency graph has a cycle"
- 第一条依赖仍然存在，未被破坏

---

## 测试 4：依赖删除（两种路径）

**4a. 按依赖容器索引删除**

**步骤：**

1. 在测试 3 的基础上
2. 选择菜单 `10` (List dependencies) 查看依赖列表
3. 确认有一条依赖从任务 0 指向任务 1
4. 选择菜单 `11` (Remove dependency by index)
5. 输入依赖索引（例如 `0`）删除该依赖
6. 再次列出依赖，确认已删除

**预期结果：**

- 依赖删除成功
- 任务下标与依赖关系正确调整
- 依赖列表为空

**4b. 按任务对（前置+后置）删除（新增）**

**步骤：**

1. 重新添加一条从任务 0 到任务 1 的 FS 依赖
2. 选择菜单 `12` (Remove dependency by task pair)
3. 输入前置任务索引 `0`、后置任务索引 `1`
4. 再次列出依赖，确认已删除

**预期结果：**

- 按任务对删除成功
- 与按索引删除行为等价
- 若任务对之间不存在依赖，返回 "Invalid argument" 错误

**4c. 边界情况验证（新增）**

**步骤（验证重复删除）：**

1. 添加一条从任务 0 到任务 1 的 FS 依赖
2. 选择菜单 `12`，输入前置 0、后置 1，删除成功
3. 再次选择菜单 `12`，输入相同的前置 0、后置 1
4. 系统应拒绝，返回 "Invalid argument" 错误

**预期结果：**

- 第一次删除成功
- 第二次删除拒绝，错误提示清晰
- 程序不崩溃，可继续操作

**步骤（验证不存在的任务对）：**

1. 建立一个包含 3 个任务且无依赖的项目
2. 选择菜单 `12`，输入前置 0、后置 2（这两个任务没有直接依赖）
3. 系统应拒绝，返回 "Invalid argument" 错误

**预期结果：**

- 错误提示明确区分"任务不存在"与"依赖不存在"

## 测试 5：资源分配与里程碑约束

**步骤：**

1. 在测试 4 的基础上（此时没有依赖，重新添加依赖以便后续验证）
2. 选择菜单 `13` (Add resource)
3. 输入资源名称（例如 `Developer`）、单位成本（例如 `100`）
4. 选择菜单 `15` (Assign resource)
5. 选择任务 0（AnalysisPhase，里程碑），资源 0，数量 1
6. 系统应拒绝，因为里程碑不可分配资源

**预期结果：**

- 资源创建成功
- 里程碑不允许分配资源，拒绝消息清晰
- 普通任务可正常分配资源（可在测试中选择其他普通任务验证）

---

## 测试 6：项目导出与重新导入一致性

**步骤：**

1. 在前面测试的基础上建立一个包含多任务、多依赖、多资源的项目
2. 选择菜单 `3` (Export PPM)
3. 输入导出路径（例如 `test_export.PPM`）
4. 选择菜单 `1` (Create project) 创建新的空项目
5. 选择菜单 `2` (Import PPM)
6. 输入刚导出的文件路径
7. 使用菜单 `18` (Show statistics) 验证导入后的数据

**预期结果：**

- 导出成功，文件可读，LastError 为空
- 重新导入后，任务数、依赖数、资源数与原项目一致
- 任务名称、工期、资源成本等详细属性完全相同
- 没有数据丢失或损坏

**6a. 检验导出数据一致性（新增）**

**步骤：**

1. 建立一个包含 2 个任务、1 条依赖、1 个资源分配的项目：
   - 任务 0：名称 "Design"，工期 5 天，分配资源 0（单位成本 100）数量 2
   - 任务 1：名称 "Development"，工期 10 天，分配资源 0 数量 3
   - 依赖：任务 0 → 任务 1（FS 类型，Lag 0）
2. 导出为 `export_verify.PPM`
3. 手工查看文件内容，确认 P、T、D、A 各行格式正确
4. 重新导入，通过菜单检查各任务的成本计算是否一致

**预期结果：**

- T 行的工期、M 行不出现、D 行的依赖类型与参数准确
- A 行的资源分配（任务索引 资源索引 数量）准确
- 导出文件可以被导入器完整还原，无数据丢失
- 任务 0 总成本 = 5 × 100 × 2 = 1000，任务 1 总成本 = 10 × 100 × 3 = 3000

## 测试 7：导出失败时捕获错误信息（新增）

本测试验证导出器的错误处理与控制器的异常映射机制。

**7a. 不可写路径导出**

**步骤：**

1. 导入或创建一个有效的项目
2. 选择菜单 `3` (Export PPM)
3. 输入不可写路径（例如 `/invalid/path/file.PPM` 或 `C:\\Program Files\\readonly\\file.PPM`，或任何当前用户无写权限的目录）
4. 程序应该拒绝导出并返回错误

**预期结果：**

- 导出失败，返回 FILE_ERROR 状态
- 错误消息准确反映失败原因（目录不存在或无权限）
- LastError 被正确更新，UI 显示错误细节
- 项目数据未被破坏，程序继续正常运行

**7b. 不支持的文件格式**

**步骤：**

1. 在导出菜单中输入扩展名非 .PPM 的文件名（例如 `output.XML` 或 `data.CSV`）
2. 程序应该拒绝导出

**预期结果：**

- 导出失败，返回 FILE_ERROR 状态
- 错误信息指出"不支持的文件格式"或类似提示
- LastError 中记录被拒绝的扩展名

**7c. 导出失败后状态保持**

**步骤：**

1. 执行一次导出失败操作（上述 7a 或 7b）
2. 选择菜单 `18` (Show statistics) 查看项目统计
3. 选择菜单 `10` (List dependencies) 查看依赖是否完整

**预期结果：**

- 导出失败不会修改项目内容
- 统计数据与之前一致，无任务/依赖/资源丢失
- 程序可继续正常操作

## 测试 9：负 Lag 支持

**步骤：**

1. 新建项目
2. 添加任务 A（工期 5 天）和任务 B（工期 3 天）
3. 添加 SS 类型依赖，前置 A，后置 B，**Lag 为 -1**
4. 进行 CPM 调度（菜单 17）

**预期结果：**

- 负 Lag 被正确解析（不报错）
- 任务 B 的 ES 应早于任务 A（因为 SS -1 允许后继提前 1 天开始）
- 调度结果反映这种提前时间关系

## 快速检查清单

完成以上 11 个主测试与子测试后，请通过以下清单进行最终验收：

- [ ] **项目管理**：新建/导入/导出无异常；导出失败时返回 FILE_ERROR，LastError 正确更新
- [ ] **任务操作**：增删改操作准确；工期修改为 0 时自动转为里程碑；下标调整正确
- [ ] **依赖管理**：支持 FS/SS/FF/SF 四种类型与负 Lag；环路检测生效
- [ ] **依赖删除**（新增）：支持按索引删除与按任务对删除两条路径；重复删除返回 INVALID_ARGUMENT；不存在的任务对返回错误
- [ ] **资源分配**：资源创建成功；里程碑不可分配资源；成本计算正确（工期 × 单位成本 × 数量）
- [ ] **CPM 调度**：项目校验通过时，总工期、关键路径、各任务的 ES/EF/LS/LF 计算准确
- [ ] **导出验证**（新增）：导出文件格式正确（P/T/D/A 行）；重新导入后数据完全一致；无数据丢失
- [ ] **错误处理**（新增）：导出到不可写路径时返回 FILE_ERROR；不支持的格式拒绝导出；错误消息清晰，LastError 正确更新；失败后项目数据保持
- [ ] **稳定性**：所有错误提示清晰；程序不崩溃；人机交互流畅
- [ ] **回归测试**：自动化测试全部通过（run_tests.ps1 ≥ 25 个用例）
